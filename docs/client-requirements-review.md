## ヒアリング結果サマリ

事前の書面要件とヒアリングで大きなギャップがあった。書面では抽象的に列挙されていた要件が、ヒアリングを経て具体的なユースケース・トリガー・スコープに変化し、また書面にない新規要件が複数浮上した。

**事前要件 vs ヒアリングで判明した実態:**

| 事前要件 | ヒアリングでの実態 | 変化 |
|----------|-------------------|------|
| ① 入力→登録用データの自動生成 | 直接議論されず。アフィリコードURL発行効率化の文脈で間接的に言及 | 優先度低下 |
| ② データ集計の自動化（API連携） | 「おまとめ率（アップセル率）」の日次自動計算が具体的ニーズ。アドエビスCV÷アフィリコード合算CV | より具体的に。スコープは狭い |
| ③ ステータス変更→Chatwork通知 | Googleドライブへのアップロード→自動解析→通知が実際のニーズ | トリガーが全く違う |
| ④ スプシ間データの集約 | 言及なし | 優先度低下 |
| ⑤ NGワードの自動検出 | テキスト＋画像（OCR）。Claude APIを使ったAI一次スクリーニング。Googleドライブ/フォームがトリガー | 難易度が大幅に上昇 |
| ⑥ 書式込みでの転記 | 言及なし | 優先度低下 |
| (新) メンテナンス・障害情報の自動通知 | アドエビス管理画面のメンテ情報をスクレイピングしChatwork通知 | 新規要件 |
| (新) Googleドライブ/フォーム連携 | ドライブに素材を入れると自動でAI解析→スプシ転記 | 新規要件 |

## 重要な新事実

- アドエビス/アフィリコードの使い分けが3パターン（アドエビスのみ / 両方 / アフィリコードのみ）
- CV乖離問題がビジネス上のペインポイント（技術では解決しない運用ポリシーの問題）
- インターン生が日常的にチェック業務を担当（UIの簡易さが重要）
- クライアントからのフィードバック（煽り表現等）は文脈依存でAI完全自動化は困難→「完全NGだけ弾く」方針
- エンジニア側から「全部アフィリコード経由に統一」の提案があったが、CV乖離で運用者が抵抗

## 各要件の詳細

### A. メンテナンス・障害情報の自動通知

- **実現可否**: ⚠️ 条件付き（利用規約確認が先決）
- **実装方法**:
  - 方法1（推奨）: アドエビスに「メンテナンス情報のメール通知またはAPI」があるか問い合わせる→あればGASでメール解析→Chatwork転送
  - 方法2: GAS UrlFetchAppでログインしてスクレイピング（MFA/reCAPTCHAがなければ可能）
  - 方法3: Puppeteer + Cloud Run で定期スクレイピング（JSレンダリング対応。ログインフォーム操作可）
- **注意点**:
  - 利用規約で自動アクセスが禁止されている可能性がある
  - まず公式の通知手段（メール通知、API、RSSフィード）を確認するのが最短経路
  - Cloud Schedulerで朝4時バッチ→Chatwork通知は技術的に確立

---

### B. おまとめ率（CV数）集計の自動化

- **実現可否**: ✅ 実現可能
- **具体的な処理フロー**:
  - アフィリコードから合算CV数を取得
  - アドエビスからおまとめ数を取得
  - 割り算してパーセントを算出
  - 前日分・当日分・月合計を毎日昼頃に自動更新
- **実装方法**:
  - API利用可能なら: GAS時間トリガー（毎日12時）→UrlFetchAppでAPI呼出し→スプシに書込み
  - API利用不可なら: 手動CSVを所定フォルダにエクスポート→GASが読込み→計算→スプシ出力
- **注意点**:
  - アドエビスはAPI対応プラン制限の可能性あり
  - アフィリコードはビジネスプラン以上が必要
  - GASの1日のURLフェッチ上限: 無料2万回/Workspace10万回（日次集計なら余裕）
  - 3パターンの場合分け（アドエビスのみ/両方/アフィリコードのみ）に対応する設計が必要

---

### C. クリエイティブNGチェックAI

- **実現可否**: ✅ 実現可能（ただし最も工数がかかる）
- **処理フロー**:
  1. Googleドライブの「一次チェック」フォルダに画像/Zipをアップロード（またはGoogleフォームで送信）
  2. GASがポーリング（5〜15分間隔）で新規ファイルを検知
  3. Zipの場合はUtilities.unzip()で解凍
  4. Claude APIのVision機能で画像内テキストを読み取り
  5. 既存NGシートのNGワードリストと照合
  6. 結果をスプレッドシートに転記（NG該当箇所をハイライト）
  7. Chatworkに通知
- **技術的ポイント**:
  - Claude Vision: 印刷文字のOCR精度は誤字率約2.1%、広告テキスト読み取りには十分
  - GASからClaude API呼出し: UrlFetchApp.fetchでPOST。APIキーはScript Propertiesに保存
  - コスト: 1クリエイティブあたり約1〜1.5円。バッチAPIで50%割引、プロンプトキャッシュで最大90%削減可
  - Googleドライブトリガー: 専用トリガーはGASにない。時間駆動ポーリング（推奨）またはDrive API Push通知（要Cloud Functions）
  - Googleフォーム: onFormSubmitトリガーで即時起動可能。インターン生の利用に最適
  - Zipファイル: Utilities.unzip(blob)で対応。上限50MB、パスワード付きは非対応
  - 重複処理防止: LockService + PropertiesServiceで処理済みファイルIDを管理
  - 6分制限: 1件なら数秒で完了。大量ファイルはPropertiesServiceでチャンク処理
- **スモールスタート案**:
  - まずGoogleフォーム経由でテキストのNGワードチェックのみ実装
  - 次に画像（Claude Vision）対応を追加
  - Googleドライブ連携は最後に

## 推奨フェーズ（更新版）

エンジニア側の提案を踏まえた優先順位:

**Phase 1（すぐ着手可能・効果実感が早い）**
- A. メンテナンス通知Bot → まずアドエビスに公式通知手段を問い合わせ。なければスクレイピング検討
- B. おまとめ率の日次自動計算 → API契約プラン確認が先決

**Phase 2（メイン開発）**
- C. クリエイティブNGチェックAI → Googleフォーム+テキストNGチェックからスモールスタート → 画像対応 → ドライブ連携

**Phase 3（必要に応じて）**
- D〜F（入力自動生成・スプシ集約・書式転記）
